# ADR 052: Routing Engine — OpenRouteService (ORS)

## Context

Office Hero needs to generate three ranked route options when dispatching a Job to a
Vehicle. This requires a mapping/routing engine that can calculate travel time and
optimal sequences given a list of stops and a current vehicle position.

## Decision

**OpenRouteService (ORS) — community API first; self-hosted ORS if latency is
unacceptable.**

## Rationale

| Factor | ORS (community) | Google Maps | Mapbox |
| ------ | --------------- | ----------- | ------ |
| Cost | Free (rate-limited) | Paid after free tier | Paid after free tier |
| Open source | Yes (can self-host) | No | No |
| Accuracy | Good (OpenStreetMap data) | Excellent | Very good |
| Rate limits | 500 req/day (community) | $200/month credit | $200/month credit |
| Self-host option | Yes (Docker) | No | No |
| Latency | Variable (community) | Fast | Fast |

500 req/day is sufficient for early launch (5 Tenants, low Job volume).
When we outgrow the community server, we self-host ORS on Fly.io (free tier
has enough CPU for a small ORS instance) before paying for Google Maps.

## Isolation Strategy

```python
class RoutingAdapter(Protocol):
    async def routing_options(
        self, origin: Coordinates, stops: list[Coordinates]
    ) -> list[RouteOption]: ...
```

`ORSAdapter` wraps the ORS HTTP API. `GoogleMapsAdapter` or `MapboxAdapter` would
implement the same protocol. Swap = new class, zero changes to services.

## Consequences

- Community ORS latency is unpredictable; routing option generation SLO is set at
  < 8 s to accommodate this
- Self-hosting ORS is a defined upgrade path (adds ~1 GB RAM; fits Fly.io)
- Route quality depends on OpenStreetMap data quality; acceptable for US coverage
  in plumbing/HVAC/pest control metro areas
