# ADR 060: Authentication — JWT RS256 + bcrypt + Refresh Tokens

## Context

Office Hero needs authentication that:

- Is stateless (no server-side session store required at launch)
- Is secure against algorithm-confusion attacks
- Supports short-lived access tokens (reduces blast radius of token theft)
- Allows token revocation for logout/account compromise
- Works for web, mobile, and MCP server clients

## Decision

**JWT RS256 (asymmetric) access tokens (15-minute expiry) + refresh tokens
(7-day expiry, stored in DB) + bcrypt for password hashing.**

## Rationale

| Factor | JWT RS256 | JWT HS256 | Session Cookies | OAuth2 only |
| ------ | --------- | --------- | --------------- | ----------- |
| Algorithm confusion protection | Yes | No | N/A | Varies |
| Stateless verification | Yes | Yes | No | No |
| Revocable access tokens | Via blocklist | Via blocklist | Native | Native |
| Mobile app compatible | Yes | Yes | Harder | Yes |
| MCP server compatible | Yes | Yes | No | Complex |
| Short-lived access window | Yes (15 min) | Yes | N/A | Yes |
| Key rotation support | Easy (RS256) | Hard | N/A | Varies |

**Why RS256 over HS256:**
RS256 (asymmetric) uses a private key to sign and a public key to verify.
If the public key is ever exposed (e.g., via a `/jwks` endpoint), tokens cannot be
forged without the private key. HS256 with a single shared secret means any service
with the secret can forge tokens. RS256 also prevents the "algorithm:none" attack
by design — the private key is never transmitted.

**Why short-lived access tokens:**
A 15-minute window limits how long a stolen access token is useful without requiring
a server-side blocklist for every request. Refresh tokens (stored in DB) can be
revoked instantly on logout or compromise.

**Password hashing — bcrypt:**
bcrypt is battle-tested, has built-in salting, and is deliberately slow (work factor).
Alternatives: Argon2id (stronger but fewer Python libs), scrypt (similar). bcrypt is
the safe default for solo-dev project at this scale.

## JWT Payload

```json
{
  "sub": "user_uuid",
  "tenant_id": "tenant_uuid",
  "role": "TenantAdmin",
  "permissions": ["jobs:write", "routes:read"],
  "iat": 1709900000,
  "exp": 1709900900,
  "jti": "unique_token_id"
}
```

- `jti` (JWT ID) enables blocklisting without full token storage
- `permissions[]` enables fine-grained permission checks beyond role (see RBAC section)
- Access token never contains password hash or sensitive data

## Token Lifecycle

```text
POST /auth/login
  → verify bcrypt(password, hash)
  → issue access_token (15 min, RS256)
  → issue refresh_token (7 days, stored in DB as hashed value)
  → return { access_token, refresh_token, token_type: "bearer" }

POST /auth/refresh
  → verify refresh_token against DB record (hashed comparison)
  → if valid and not revoked: issue new access_token
  → optionally rotate refresh_token (sliding window)

POST /auth/logout
  → mark refresh_token as revoked in DB
  → (access_token expires naturally in < 15 min)
```

## Isolation Strategy

**Auth is fully isolated in the middleware and a dedicated auth service:**

- `src/office_hero/api/middleware/auth.py` — JWT decode, claim extraction
- `src/office_hero/services/auth_service.py` — login, refresh, logout, bcrypt ops
- `src/office_hero/repositories/token_repository.py` — refresh token storage/revocation
- `src/office_hero/api/schemas/auth.py` — login/refresh request/response schemas

No router, service, or repository outside `auth_service.py` calls `bcrypt` or
decodes JWTs. They only receive the decoded claims object from middleware.

**To replace JWT with another auth mechanism (e.g., Paseto, opaque tokens):**
Replace `auth.py` middleware and `auth_service.py` — all other layers are unchanged.
The claims object interface (what the service layer receives) stays the same.

**To swap bcrypt for Argon2id:**
Change the `AuthService.hash_password()` and `verify_password()` methods only.

## Consequences

- RS256 requires a key pair: generate at deploy time, store private key in Fly.io
  secrets, publish public key at `GET /auth/.well-known/jwks.json` (for MCP/external)
- `refresh_tokens` table: `id`, `user_id`, `token_hash`, `expires_at`, `revoked_at`
- Rate limiting on all auth endpoints (see OWASP A07 section in HLD):
  `/auth/login`, `/auth/refresh` — 10 req/min per IP
- `python-jose[cryptography]` or `PyJWT` (+ `cryptography` extra) for RS256
- bcrypt work factor = 12 (balance between security and < 500 ms login response)
