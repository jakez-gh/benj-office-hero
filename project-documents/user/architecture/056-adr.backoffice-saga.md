# ADR 056: Back-Office Integration Risk Mitigation — Saga + Outbox Pattern

## Context

Office Hero will integrate with external back-office systems (ServiceTitan, PestPac,
Jobber, etc.). Some operations span both Office Hero and the external system.
If steps succeed partially, data can become inconsistent in ways that are hard to
detect and expensive to repair. Examples:

- Create Customer in Office Hero **succeeds**, but create Customer in ServiceTitan
  **fails** → Office Hero has a Customer the back-office system does not know about.
- Invoice generated in PestPac **succeeds**, but Job status updated in Office Hero
  **fails** → billing and field operations are out of sync.
- Route dispatched in Office Hero **succeeds**, but push notification to technician
  app **fails** → Technician is unaware of assignment.

These are **distributed transaction** problems. Two-phase commit (2PC) across HTTP
APIs is not practical; it introduces tight coupling and is fragile.

## Decision

**Saga pattern** for multi-step cross-system operations, combined with the
**Transactional Outbox pattern** for reliable event publishing.

## Rationale

### Why not 2PC

2PC requires all participants to implement a prepare/commit/rollback protocol.
External SaaS APIs (ServiceTitan, PestPac) do not expose this. Even if they did,
holding locks across HTTP calls creates deadlock risk and long-duration transactions.

### Saga Pattern

A Saga is a sequence of local transactions where each step has a **compensating
transaction** that can undo its effect if a later step fails.

```text
Step 1: Create Customer in Office Hero DB       → compensate: delete Customer
Step 2: Create Customer in ServiceTitan API     → compensate: delete via ST API
Step 3: Link ST Customer ID to OH Customer      → compensate: clear the link
```

If Step 3 fails, compensate Step 2 then Step 1. Each step and its compensating
action are independently atomic.

**Choreography vs Orchestration:**

- We use **orchestration** (a Saga Orchestrator service/class coordinates steps)
  rather than choreography (event-driven). Orchestration is easier to reason about,
  debug, and test for a solo developer. The orchestrator lives in the service layer.

### Transactional Outbox Pattern

To reliably trigger Saga steps without losing events on crash:

```text
┌──────────────────────────────────────────┐
│  Office Hero DB Transaction              │
│  1. Write domain record (e.g. Customer)  │
│  2. Write outbox event row               │
└──────────────────────────────────────────┘
         ↓ (background worker)
┌──────────────────────────────────────────┐
│  Outbox Relay (async worker)             │
│  Reads unprocessed outbox events         │
│  Calls next Saga step                    │
│  Marks event processed (or dead-letter)  │
└──────────────────────────────────────────┘
```

The domain write and outbox write happen in one DB transaction. The outbox relay
runs outside that transaction — it can be retried safely without risking double writes
to the domain record.

### Idempotency Keys

Every external API call includes an idempotency key (`X-Idempotency-Key` header where
supported, or a unique field in the request body). This makes all Saga steps safe to
retry. The key is stored in the outbox event and reused on retry.

### Dead-Letter Handling

Failed Saga steps that exhaust retries are moved to a dead-letter table. A
`GET /admin/dead-letters` endpoint (Operator-only) surfaces these for manual
resolution. Monitoring alerts on any dead-letter growth.

## Ordering Risk: Specific Scenarios

### Scenario A: Create Customer with Back-Office Sync

```text
1. Begin OH DB transaction
2. INSERT customers (status: syncing)
3. INSERT outbox_events (type: customer_created, payload: {...})
4. Commit transaction
--- transaction boundary ---
5. Outbox relay: POST /customers to external API
6. Outbox relay: UPDATE customers SET external_id=..., status: synced
7. Mark outbox event: processed
```

Crash between 4 and 5: relay retries step 5 on next run.
Crash between 5 and 6: idempotency key prevents duplicate at external API.

### Scenario B: Dispatch with External Work Order Creation

Some back-office systems require a Work Order (their term for Job) to exist before
accepting a dispatch. Steps must be:

```text
1. OH: Create Job (status: pending)
2. External: Create Work Order → get external_id
3. OH: Link external_id to Job
4. OH: Dispatch (update Route, RouteStops)
5. External: Update Work Order status to Dispatched
```

If Step 5 fails → retry (idempotent).
If Step 2 fails → compensate Step 1 (mark Job failed, surface to dispatcher).
**Never proceed to Step 4 without Step 3 succeeding** — the route should not
be committed until both systems have the Job/Work Order.

This ordering constraint is enforced by the Saga orchestrator checking each step's
result before proceeding.

## Implementation Guidance

- The `BackOfficeAdapter` protocol includes a `health_check()` method; the Saga
  checks it before beginning any multi-step operation. If the external system is
  unreachable, the operation is queued (not failed immediately) with exponential
  backoff.
- Saga state (current step, idempotency key, compensate history) is persisted in the
  `saga_log` table — not in application memory. This survives process restarts.
- Unit tests for each Saga inject a mock `BackOfficeAdapter` and simulate failures
  at each step to verify compensation logic.
- Integration tests run against a stub/sandbox environment for each back-office system.

## Data Model Additions

```sql
-- Outbox events (transactional outbox pattern)
CREATE TABLE outbox_events (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    tenant_id   UUID NOT NULL REFERENCES tenants(id),
    event_type  TEXT NOT NULL,
    payload     JSONB NOT NULL,
    idem_key    UUID NOT NULL DEFAULT gen_random_uuid(),
    status      TEXT NOT NULL DEFAULT 'pending',  -- pending | processing | done | dead
    attempt_count INT NOT NULL DEFAULT 0,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
    processed_at TIMESTAMPTZ
);

-- Saga execution log
CREATE TABLE saga_log (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    tenant_id   UUID NOT NULL REFERENCES tenants(id),
    saga_type   TEXT NOT NULL,
    current_step INT NOT NULL DEFAULT 0,
    status      TEXT NOT NULL DEFAULT 'running',  -- running | done | compensating | failed
    context     JSONB NOT NULL,  -- step inputs/outputs for compensation
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

RLS policies on both tables use the standard `tenant_id` pattern.

## Consequences

- Back-office slices (24–27) must implement the Saga orchestrator for their adapter,
  not just a simple sync call.
- Foundation slice 2 (Database foundation) must create `outbox_events` and `saga_log`
  tables.
- Adds ~1 Effort point to each back-office integration slice (already rated 4/5).
- Dead-letter monitoring is part of Slice 30 (Monitoring & alerting).
- **NativeAdapter** (Office Hero as SoR) is unaffected — all operations are local
  DB transactions with no distributed risk.
