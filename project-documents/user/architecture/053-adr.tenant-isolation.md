# ADR 053: Tenant Isolation — PostgreSQL Row-Level Security

## Context

Office Hero is a multi-tenant SaaS. No Tenant may read or write another Tenant's
data. The Operator can see all data. This is a hard security requirement.

Two common approaches: **application-layer filtering** (WHERE tenant_id = ?) and
**database-layer RLS** (PostgreSQL enforces it automatically).

## Decision

**PostgreSQL RLS as the primary gate, with application-layer checks as defense-in-depth.**

## Rationale

**Application-layer only** is fragile: a single missing `WHERE tenant_id = ?` clause
leaks data. In a complex codebase with AI-assisted development, such an omission is
plausible.

**RLS** makes the database enforce isolation automatically. A missing clause in the
application code doesn't leak data — the DB rejects it. It is a far stronger guarantee.

**Defense-in-depth:** The service layer also validates that the `tenant_id` in every
write/read matches the JWT claim. Two independent layers both must be bypassed to leak
data.

## Implementation

```sql
-- Session variable set by API middleware from JWT before every query:
SET LOCAL app.tenant_id = '<tenant_uuid>';

-- Policy on every tenant-scoped table:
CREATE POLICY tenant_isolation ON jobs
    USING (tenant_id = current_setting('app.tenant_id')::uuid);
```

Operator queries bypass RLS via a separate DB role with `BYPASSRLS` (set only
when role = Operator is confirmed in JWT).

Alembic migrations manage all RLS policy creation — policies are version-controlled
alongside the schema.

## Consequences

- All tenant-scoped tables require a `tenant_id` column and an RLS policy
- SQLAlchemy sessions must set `app.tenant_id` before executing queries
  (handled by a session factory in `db/session.py`)
- Unit tests run against mock repositories (no RLS complexity in unit tests)
- Integration tests use a real Neon branch with RLS policies applied
