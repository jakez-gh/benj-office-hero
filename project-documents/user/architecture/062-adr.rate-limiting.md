# ADR 062: Rate Limiting — global, tiered, and runtime-configurable

## Context

Rate limiting is required to prevent abuse, brute-force attacks, and accidental
flooding. Earlier documents specify use of `slowapi` middleware applied to all
endpoints with tiered limits (auth, writes, reads, global fallback). The
requirement has now expanded:

* Operators must be able to adjust rate limits live via the admin UI without
  restarting the server.
* They must also be able to block/unblock requests by filters such as account,
  tenant, IP, or geographic region.

## Decision

Continue using `slowapi` for its Starlette/ASGI compatibility but add a runtime
configuration layer backed by the database. The rate limiter will consult the
DB on each request so changes take effect immediately.

## Rationale

| Option | Pros | Cons |
|--------|------|------|
| Hard-coded limits in code | Simple | Requires redeploy to change; not manageable by Operators |
| Configuration file reloaded occasionally | Slightly flexible | Still needs reload, might miss spikes |
| DB-backed limits check per request | Fully dynamic; Operators can modify live | Slight overhead (~1ms) per request |

The additional overhead is minimal compared to network and routing latency, and
it satisfies the requirement for live control.

## Implementation

1. Create a new table, `rate_limits`, with columns `name`, `limit`, `per_seconds`,
   `scope` (global, tenant, user, ip, geo, custom), and `filters` (JSONB).
2. A `RateLimitManager` service queries this table (cached for 1s) to build a
   `slowapi` `Limiter` instance.
3. The FastAPI middleware uses `async def current_limit(request):` hook to fetch
   the appropriate limit from `RateLimitManager` on each request.
4. Admin UI pages allow Operators to create/modify/delete rows in
   `rate_limits`—changes take effect immediately thanks to the cache expiry.
5. Special `ban_list` table stores filters to block (same schema as `rate_limits`
   but with `limit=0`). The middleware checks this table first and rejects
   matching requests with 403.

### Example record

```sql
INSERT INTO rate_limits (name, limit, per_seconds, scope, filters)
VALUES ('auth_login', 10, 60, 'global', '{}');
```

### Filter examples

* `{"tenant_id": "..."}` — override limit for a specific tenant
* `{"ip": "203.0.113.0/24"}` — stricter limit for a subnet
* `{"geo": "CN"}` — block or throttle by country code

## Consequences

* Operators need UI components for editing rate limits; these are slices 5a and
  later dispatch slices will reference them.
* RateLimitManager cache expiry should be tunable (default 1s) to balance
  dynamism vs DB load.
* Performance regression expected to be <2% at 1000 RPS.
* `ban_list` entries behave like explicit deny ACLs and are checked before all
  other limits.

**This ADR documents the pattern of a runtime-configurable, DB-driven
rate-limiter.**
