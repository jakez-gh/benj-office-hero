# ADR 051: Web Framework — FastAPI

## Context

Office Hero needs a Python web framework for its REST API. The API is the single
interface consumed by all clients (Tenant Admin web, Technician web, Android app, MCP
server). Candidates: **FastAPI** and **Flask**.

## Decision

**FastAPI.**

## Rationale

| Factor | FastAPI | Flask |
| ------ | ------- | ----- |
| Async support | Native (ASGI) | Requires extensions |
| OpenAPI docs | Auto-generated from type hints | Manual or extension |
| Input validation | Pydantic built-in | Marshmallow/manual |
| Type safety | End-to-end via Pydantic | Opt-in |
| MCP compatibility | OpenAPI spec → MCP auto-generation | Possible but more work |
| Performance | Higher (async I/O for ORS + DB calls) | Lower (sync default) |
| Learning curve | Moderate | Low |

FastAPI's auto-generated OpenAPI spec is particularly valuable: the MCP server can be
generated from it, and external integrators get accurate docs automatically.

## Isolation Strategy

All HTTP concerns live in `src/office_hero/api/`. Services and repositories have no
FastAPI imports. Swapping to Flask or another framework means replacing `api/` only.

## Consequences

- All request/response schemas are Pydantic models (also used for config via
  `pydantic-settings`)
- Async requires `asyncpg` or `async` SQLAlchemy sessions — slightly more setup
  than sync SQLAlchemy, but fully supported
- Testing uses `httpx.AsyncClient` + `pytest-asyncio`
