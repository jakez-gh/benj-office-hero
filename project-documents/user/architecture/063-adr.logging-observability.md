# ADR 063: Logging & Observability — structlog, audit log, Grafana/Loki

## Context

Our system needs robust logging for debugging, performance monitoring, and
security forensic analysis. Earlier documents mention `structlog` and an audit
log, but a formal ADR is required to explain the pattern and ensure any future
changes follow the same approach.

## Decision

Use the **Structured Logging** pattern with `structlog` for application logs and a
separate **Append-Only Audit Log** table for security events. Ship logs to a
central aggregator (Grafana Loki or Better Stack) and build dashboards there.

## Rationale

Structured logs (JSON records) are much easier to search, filter, and parse
than plaintext. They allow correlation across services by including fields like
`request_id` and `tenant_id`. An audit log is separate because security events
must not be deletable or alterable by application code; they require tighter
controls and longer retention.

Grafana Loki (open source) integrates nicely with Grafana dashboards; Better
Stack is a hosted alternative with a generous free tier. Both ingest via
HTTP or syslog from Fly.io.

## Pattern Details

### Structured Logging

* Use `structlog` configured for JSON output.
* Middleware injects `request_id` (UUID) and `tenant_id` into context.
* All loggers created via `structlog.get_logger()` so context flows automatically.
* Example log line:

```json
{"timestamp": "2026-03-08T14:00:00Z", "level": "info", "event": "job.created", "job_id": "...", "tenant_id": "...", "request_id": "...", "duration_ms": 23}
```

### Audit Log

* Separate table `audit_events` with columns:
  `id, timestamp, tenant_id, user_id, event_type, details JSONB`.
* Database trigger prevents `UPDATE` or `DELETE`; only `INSERT` allowed.
* Application writes audit records via a simple `AuditService` with method
  `log_event(event_type, details)`.
* Audit records are **never** shipped to Loki; they are queried directly by the
  Operator UI (filterable by `event_type`, `user_id`, `tenant_id`, time range).
* Audit service runs in async background to avoid blocking request path.

### Observability Platform

* Fly.io logs forwarded to Loki via Vector (Fly.sh log drain). Configure
  `loki.push_url` and API key in Fly.io secrets.
* Grafana Cloud free tier hosts the dashboards; Loki as the data source.
* Use pre-built dashboards for FastAPI/Starlette (Grafana marketplace).
* Add custom panels for:
  * Uptime, response time, error rate, active tenants, route latency.
  * Dead-letter queue depth (from audit or separate table).
* Configure Grafana alert rules for anomalies (error rate spikes, dead-letter >0).

## Consequences

* Developers must compose logs with structured fields; avoid plain string
  concatenation.
* `requirements.txt` or pyproject includes `structlog`, `python-json-logger`.
* Audit log table schema is version-controlled via Alembic.
* UI for audit log queries must be built (see Operator dashboard slice).
* Onto microservices if we split later: continue pattern per service for
  consistent correlation IDs.

This ADR documents both the logging pattern and the observability platform
choice, fulfilling the requirement that every referenced pattern have full
explanation and rationale.
