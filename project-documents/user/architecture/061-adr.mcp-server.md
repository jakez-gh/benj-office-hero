# ADR 061: MCP Server — Python MCP SDK + OpenAPI Auto-generation

## Context

Office Hero needs an MCP (Model Context Protocol) server to expose API capabilities
to AI assistants (Claude, etc.). Requirements:

- Same RBAC and auth as the REST API (no separate permission model)
- Cover Job entry, dispatch, routing, customer lookup as MCP tools
- Minimise maintenance burden — must not require manual sync with the API

## Decision

**Python MCP SDK (`mcp` package) as a separate Fly.io process, with tools
auto-generated from the FastAPI OpenAPI spec where possible and hand-written
for complex operations (dispatch, routing option selection).**

## Rationale

| Factor | Python MCP SDK | TypeScript MCP SDK | Manual HTTP wrapper |
| ------ | -------------- | ------------------ | ------------------- |
| Language match | Yes (Python) | No (extra language) | Yes |
| OpenAPI → tool generation | Via codegen/custom | Via codegen | Manual |
| Auth reuse | Same JWT middleware | Separate | Separate |
| Maintenance overhead | Low (auto-gen) | Medium | High |
| Official SDK | Yes (Anthropic) | Yes (Anthropic) | N/A |

**Why auto-generate from OpenAPI:**
FastAPI auto-generates an OpenAPI spec at `/openapi.json`. A build-time script
reads this spec and generates MCP tool definitions (name, description, input schema,
HTTP call). This means: add a new API endpoint → it appears in MCP automatically.
Only complex multi-step operations (dispatch flow, routing selection) need hand-written tools.

**Why separate process:**

- Independent scalability
- Failure in MCP server does not affect the REST API
- Can be deployed/updated independently
- Different resource requirements (MCP sessions may be long-lived)

## Tool Categories

**Auto-generated from OpenAPI:**

- `get_customers` — list/search customers
- `get_customer` — single customer
- `create_customer` — new customer + location
- `get_jobs` — list/filter jobs
- `get_job` — single job detail
- `create_job` — new job entry
- `get_routes` — daily route view
- `get_vehicles` — vehicle list

**Hand-written (complex operations):**

- `get_routing_options` — calls `POST /jobs/{id}/routing-options`, formats 3 options
- `dispatch_job` — calls `POST /jobs/{id}/dispatch` with option selection logic
- `get_health` — platform health summary

## Isolation Strategy

**The MCP server is a pure HTTP client of the REST API — it has no direct DB access:**

```text
AI Assistant → MCP Protocol → MCP Server → HTTP → REST API → Business Logic → DB
```

- MCP server imports: `mcp` SDK, `httpx` (async HTTP client), `pydantic` (schemas)
- MCP server never imports: SQLAlchemy, FastAPI, any service or repository class
- MCP server authentication: passes the AI's access token through to the REST API
  (the AI must authenticate and obtain a JWT first)

**To replace the MCP server implementation:**
The REST API is unchanged. Replace `mcp-server/` directory with any other
MCP-compatible server (TypeScript, different framework). The OpenAPI spec remains
the source of truth.

**To replace MCP with another AI protocol:**
Only the `mcp-server/` process changes. The REST API and its OpenAPI spec are
the stable interface.

## Code Structure

```text
mcp-server/
├── pyproject.toml          # Separate package, own deps
├── src/
│   └── office_hero_mcp/
│       ├── main.py         # MCP server entry point
│       ├── tools/
│       │   ├── generated/  # Auto-generated from OpenAPI (do not edit)
│       │   └── custom/     # Hand-written complex tools
│       └── client.py       # httpx client wrapper (auth, base URL)
└── scripts/
    └── generate_tools.py   # Reads OpenAPI spec, generates tools/generated/
```

`generate_tools.py` runs as part of CI whenever the OpenAPI spec changes.

## Consequences

- MCP server deployed as a second Fly.io machine (within free tier: 3 VMs shared)
- `mcp` Python package: `pip install mcp` (Anthropic official SDK)
- `httpx[asyncio]` for async HTTP calls to the REST API
- CI step: if OpenAPI spec changes → regenerate tools → diff must be committed
- Auth: MCP server receives a JWT from the calling AI; passes it as
  `Authorization: Bearer <token>` to all REST API calls
- MCP server config: `REST_API_BASE_URL`, `MCP_SERVER_HOST`, `MCP_SERVER_PORT`
  all from environment variables
