# ADR 055: Frontend — React (web) + React Native Expo (mobile)

## Context

Office Hero requires:

- Tenant Admin web GUI (complex: dispatch dashboard, Job management, Route visualisation)
- Technician web view (lighter: own Route, Job entry)
- Technician Android app (location tracking, route viewing, offline resilience)
- Possible Tenant Admin mobile app in future
- All GUIs consume the same REST API

## Decision

**React (TypeScript) for web. React Native (Expo) for mobile.**

## Rationale

### Why React for web

| Factor | React | HTMX + Alpine | Server-rendered (Jinja2) |
| ------ | ----- | ------------- | ------------------------ |
| Dispatch dashboard UX | Excellent (rich interactions) | Limited | Poor |
| Real-time map updates | Native (state updates) | Clunky | Not feasible |
| Code sharing with mobile | Yes (React ecosystem) | No | No |
| TypeScript support | Excellent | Partial | N/A |
| Offline capability | PWA possible | Difficult | No |

The dispatch dashboard needs drag-and-drop route reordering, live Vehicle positions
on a map, and instant feedback on dispatch actions. Server-rendered templates
cannot deliver this UX without heavy custom JavaScript — effectively reimplementing
React without the ecosystem.

### Why React Native Expo for mobile (not Flutter, not native Android)

| Factor | React Native Expo | Flutter | Native Android |
| ------ | ----------------- | ------- | -------------- |
| iOS extension | Yes (same codebase) | Yes | Separate project |
| Code sharing with web | Yes (API client, types) | No (Dart) | No |
| Solo developer cost | Low | Medium | High (two codebases) |
| GPS background tracking | Yes | Yes | Yes |
| Expo managed workflow | Yes (no native build tools needed initially) | No | N/A |
| OTA updates | Yes (Expo Updates) | Partial | No |

Expo lets us build and test the Android app without Xcode or Android Studio initially
(Expo Go for development, EAS Build for production APK). Adding iOS later requires
no new language — just EAS Build for iOS.

## Shared Code Structure

```text
packages/
├── api-client/     # TypeScript API client (shared: web + mobile)
├── types/          # Shared TypeScript types matching API schemas
└── ui-components/  # Shared non-visual logic (validation, formatting)

apps/
├── admin-web/      # Tenant Admin React SPA
├── tech-web/       # Technician React web view
└── tech-mobile/    # React Native Expo (Android first)
```

A monorepo (pnpm workspaces or Turborepo) manages the packages.
This is the JavaScript portion of the project — Python remains in `src/`.

## Consequences

- TypeScript throughout the frontend (not JavaScript) — type-safety aligns with
  the project's quality-first philosophy
- Tenant Admin mobile app starts as responsive web (React SPA on mobile browser);
  native app added as a future slice if web UX is insufficient
- React Native Expo managed workflow is used until native modules require ejecting
- Background location in Expo requires `expo-location` with background mode —
  tested early (Foundation slice) as it requires Android permission configuration
