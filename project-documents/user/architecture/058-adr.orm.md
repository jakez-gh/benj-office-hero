# ADR 058: ORM — SQLAlchemy 2.x + Alembic

## Context

Office Hero needs a database access layer for Python. The API is async (FastAPI/ASGI).
Data access must be:

- Isolated from business logic (Repository pattern)
- Async-capable (non-blocking I/O for performance)
- Migration-managed (schema changes tracked and reproducible)
- SQL-injection-safe

## Decision

**SQLAlchemy 2.x (async) + Alembic for migrations.**

## Rationale

| Factor | SQLAlchemy 2.x | Tortoise ORM | Raw asyncpg |
| ------ | -------------- | ------------ | ----------- |
| Async support | Native (2.x) | Native | Native |
| Migration tooling | Alembic (mature) | Aerich (young) | Manual |
| Type hints | Full (2.x mapped attrs) | Good | N/A |
| Query flexibility | High (Core + ORM) | Medium | Highest |
| Multi-DB support | Yes (swap dialect) | Limited | PostgreSQL only |
| Ecosystem maturity | 20+ years | < 5 years | < 5 years |
| Solo dev familiarity | High (most common) | Low | Medium |

SQLAlchemy 2.x (the "v2 style" with `mapped_column`, `Mapped[T]`) adds:

- Full type-safe ORM mapping (replaces `Column()` with `Mapped[type]`)
- Native `AsyncSession` with `asyncpg` driver
- Cleaner relationship syntax
- Better IDE integration

## Isolation Strategy

**The Repository pattern fully isolates SQLAlchemy from business logic:**

```text
Service Layer → RepositoryProtocol (ABC) → ConcreteRepository → SQLAlchemy → DB
```

- Services import only `RepositoryProtocol` (an ABC) — no SQLAlchemy imports
- `ConcreteRepository` is the only layer that imports `AsyncSession`, `select`, etc.
- Unit tests inject `MockRepository` — no SQLAlchemy, no DB, instant execution
- Integration tests use real repositories against a Neon branch

**To replace SQLAlchemy:**
Implement new concrete repository classes. Service layer is unchanged.
Alembic migration files can be regenerated from the schema.

**To replace PostgreSQL with another DB:**
Swap the SQLAlchemy dialect (`asyncpg` → another driver). Repository implementations
may need minor adjustments for DB-specific features (e.g., RLS, JSONB).

## Consequences

- `AsyncSession` requires `asyncpg` as the driver (in `DATABASE_URL`)
- All DB access must use `async with session:` context managers
- Alembic auto-generates migrations from model changes; developer reviews before applying
- Alembic's `env.py` configures RLS policies as part of migration (see ADR 053)
- `pytest-asyncio` + async session fixtures required for integration tests
- No raw SQL in service or router layers; raw SQL in repositories only
  when ORM query is impractical, and only with parameterized queries
